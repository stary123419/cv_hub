# ENGINEERING NOTES

## 1. Общая идея проекта

CV Hub — это не просто сайт-резюме.

Это попытка рассматривать профессиональное представление как инженерную систему.

Основная концепция:

> Resume as Code.

Резюме должно быть:
- версионируемым
- воспроизводимым
- автоматизируемым
- независимым от платформ

---

## 2. Архитектурные решения

### Почему Astro

Astro выбран как статический генератор с минимальным runtime.

Основные причины выбора:

- Static-first архитектура (рендер на этапе сборки, а не в браузере)
- Нулевой client-JS по умолчанию
- Предсказуемый build-output
- Высокая производительность без дополнительной оптимизации
- Простая интеграция с GitHub Pages

#### Почему не React SPA

React отлично подходит для интерактивных приложений.

Но для CV Hub:
- Не требуется сложное состояние
- Нет динамического API
- Нет необходимости в hydration всего приложения

SPA увеличивает размер бандла и сложность без архитектурной необходимости.

Astro позволяет использовать React-компоненты точечно при необходимости, без превращения всего проекта в SPA.

#### Почему не Angular

Angular — полноценный application framework.

Для данного проекта это избыточно:
- Большой runtime
- Сложная конфигурация
- Высокий порог входа
- Избыточная архитектура для двух статических страниц

CV Hub сознательно строится как минималистичная система, а не как полноценное веб-приложение.

---

### Почему всего 2 страницы

- Главная (CV)
- Showcase (проекты)

Цель — минимизировать когнитивную нагрузку.

Рекрутеру не нужно 12 разделов.
Одна страница — кто я.
Вторая — что я сделал.

---

### Почему YAML как источник данных

YAML выбран как удобный человекочитаемый формат.

Преимущества:
- Простота редактирования
- Структурированность
- Легко валидировать
- Легко парсить

YAML выступает как Single Source of Truth.

Всё остальное генерируется из него автоматически.

---

### Почему URL-параметр для тем, а не localStorage

Темы переключаются через `?theme=name` в URL — без JavaScript-хранилища.

Причины:
- URL — shareable: можно поделиться конкретной темой как ссылкой
- Нет зависимости от браузерного состояния
- Работает при первом открытии без flash of unstyled content
- Прозрачно и предсказуемо

Доступные значения: `frosted`, `light`, `nordic`, `peachy`

## 3. Модель данных

Структура CV намеренно плоская и читаемая.

Принципы:
- Ясные секции (name, title, summary, skills, experience)
- Минимальная вложенность
- Отсутствие логики в данных
- Только декларативная информация

Данные отделены от представления.

```
YAML → Astro → HTML (сайт)
YAML → docx  → DOCX + TXT (скачиваемые файлы)
YAML → Playwright → PDF (скачиваемый файл)
```

Дополнительно поддерживается импорт из JSON Resume:

```
JSON Resume → convert → YAML → всё вышеперечисленное
```

## 4. Проблемы, которые решает проект

### Дублирование резюме

Обычно существуют:
- PDF версия
- DOC версия
- Веб-версия
- LinkedIn версия

Каждая редактируется отдельно.

Здесь — один источник данных.

---

### Рассинхронизация форматов

Изменение в PDF не отражается на сайте.

CV Hub устраняет это через автоматическую генерацию из YAML.

---

### Vendor lock-in

Tilda, Notion, Canva и другие инструменты ограничивают контроль.

Проект полностью статический и независимый.

## 7. Автоматизация

### Реализовано

| Инструмент | Что делает |
|---|---|
| `docx` | Генерация DOCX и TXT из YAML |
| `Playwright` | Генерация PDF из YAML через HTML-шаблон |
| `ajv` | Валидация JSON Resume по схеме |
| GitHub Actions | CI/CD: сборка + деплой на каждый push в main |
| `resume:import` | Конвертация JSON Resume → YAML |

### CLI-инструменты

```bash
npm run resume:import -- docs/cv_en.json en   # JSON Resume → YAML
npm run resume:import:all                     # оба языка сразу
npm run resume:pdf                            # YAML → PDF
npm run resume:generate                       # YAML → DOCX + TXT
npm run build                                 # всё + сайт
```

### Roadmap

- Валидация схемы YAML через Zod
- Rust-парсер YAML как отдельный эксперимент
- Поддержка нескольких профилей (DevOps / GameDev)
- JSON export для рекрутеров (публичный endpoint)

## 8. CI / GitHub Actions

CI pipeline запускается на каждый push в `main`.

```
push → main
  ↓
npm ci
  ↓
playwright install chromium
  ↓
npm run build
  ├── generate-resume.js  → DOCX + TXT
  ├── resume-export-pdf   → PDF (через Playwright)
  └── astro build         → статический сайт
  ↓
upload artifact → deploy → GitHub Pages
```

Цели CI:
- Исключить ручную сборку
- Гарантировать воспроизводимость
- Делать каждый коммит потенциальным релизом
- Поддерживать immutable build-подход
